# Stage Scenes

## Decision
Introduce a per-stage "scene" (location + immediate context) as a first-class
concept. Each of the 20 seeded stages gets an `initial_scene_name` +
`initial_scene_description`. After that, scene changes are driven by a
server-side OpenRouter classifier that inspects every new dialogue or twist
event and decides if the scene meaningfully moved. Scene changes are stored as
`stage_events` rows of type `scene_change` (enum value already existed); the
"current scene" is derived by reading the most recent `scene_change` for the
stage, falling back to the seeded initial scene. Scene context is shown in
the dialogue panel as a compact, expandable chip above the current line, and
each new scene triggers a ~5s on-stage overlay that displays the full
description before settling back to the chip.

## Context
Until now, every stage was a single, undifferentiated location. Dialogue and
twists could push the story into a kitchen, a cellar, a spaceship — but the
viewer had no way to see "where we are now," and the agents had no shared,
durable notion of physical context to ground their next line in.

## Alternatives considered
1. **Stage-level scene only.** Add `initial_scene_*` columns and stop there;
   no runtime changes. Cheapest, but immediately stale.
2. **Human-driven scene field on the twist form.** Optional field on the twist
   submission that lets the director relocate the action. Cheap and explicit,
   but most twists wouldn't carry one, and dialogue lines that should change
   the scene ("we're already at the gas station") would never change it.
3. **Agent-driven scene change via a new MCP tool (`etc_scene_set`).** Have
   the agent decide and call a new API. Requires agent runtime to understand
   scenes; cross-agent coordination ambiguous (who's "the director"?).
4. **Server-side LLM classifier on the write path.** Every dialogue/twist
   POST runs a small JSON-output classifier; if it says "changed", we append a
   `scene_change` event. *(chosen)*
5. **Per-scene AI background image** generated by Recraft on each change.
   Visually rich but expensive and slow; punted.

## Reasoning
- Option 4 keeps the agent contract unchanged. Any agent that can speak gets
  the benefit immediately.
- Scenes naturally fit the existing append-only `stage_events` log; the
  `scene_change` enum value was already present. No new table needed.
- OpenRouter unlocks cheap reasoning models (default `openai/gpt-5-nano`,
  override via `OPENROUTER_SCENE_MODEL`). At ~$0.0002/call this is sustainable.
- Doing the classifier inline (with a 4s timeout) means the SSE stream sees
  the scene_change event immediately after the source event, so viewers get a
  coherent "twist → scene change → next line" cadence without fire-and-forget
  pitfalls (Netlify killing the function before the LLM call returns).
- Text-only scene rendering keeps the UI fast and the design within the
  existing typographic system (mono label + display-font name + mono
  description). The 5s overlay animation gives the change real weight
  without locking the user out of the stage.

## Trade-offs accepted
- **Latency.** Every dialogue/twist POST gets ~0.5–1.2s of additional latency
  from the classifier round-trip. Acceptable for now; agents poll, not
  interactive humans.
- **Cost per event.** ~$0.0001–$0.0003 per event. Tracked; if usage grows,
  we can either downgrade the model further or move to fire-and-forget
  with `waitUntil` on Netlify.
- **False negatives bias.** The classifier prompt is tuned to *stay* on the
  current scene unless the location/context clearly shifted. We will miss
  some legitimate scene changes; we tolerate this to avoid scene-thrashing.
- **No backfill.** Existing stage event histories don't get retroactive scene
  reconstruction. New events from this point forward grow the scene log.
- **No per-scene image.** Stage backdrop stays the same regardless of scene.
  Revisit if/when an in-house cheap image pipeline exists.
- **Single-source-of-truth on write path.** If the classifier is slow/down,
  the dialogue still posts, just without a scene update.

## How to revisit
- If the model proves wrong too often, replace the prompt or swap the model
  via `OPENROUTER_SCENE_MODEL`.
- If latency on the write path becomes a concern in production, move the
  classifier behind a queue / Netlify `waitUntil` and live with a 1–2s lag
  in SSE.
- If image-per-scene is wanted later, add a column on `scene_change.content`
  for `imageUrl` and a separate generator job; the UI overlay is already
  ready to host an image.
